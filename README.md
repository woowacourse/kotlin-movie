# 🚀 1단계 - 영화 예매

## 1. 기능 요구 사항

### 1.1 상영 편성

- 하나의 영화는 상영 가능 기간 내에서 하루 여러 차례 상영될 수 있다.
- 상영 시간은 영화 상영 길이와 상영관 운영 시간을 기준으로 결정된다.
- 하나의 영화가 여러 상영관에서 동시에 상영되는 경우는 고려하지 않는다.

### 1.2 예매

- 사용자는 영화와 상영 시간을 선택한 뒤 좌석을 선택한다.
- 여러 영화를 한 번에 예매할 수 있다.
- 한 번에 예매하는 상영들 간 시간이 겹치면 함께 예매할 수 없다.

### 1.3 좌석

- 좌석 표기는 행(알파벳) + 열(숫자) 형식이다. 예: A1, C3, E4
- 이미 예약된 좌석은 다시 선택할 수 없다.
- 좌석 등급별 가격이 다르다.

### 1.4 좌석 등급 가격

| 등급 | 가격 |
  |---|---:|
| S석 | 18,000원 |
| A석 | 15,000원 |
| B석 | 12,000원 |

### 1.5 할인 및 결제

- 무비데이 할인: 매월 10일, 20일, 30일 상영 시작 건은 10% 할인
- 시간 할인: 오전 11시 이전 또는 오후 8시 이후 시작 상영은 2,000원 할인
- 포인트: 예매 금액에서 직접 차감
- 결제 수단 할인

  | 수단 | 할인율 |
  |---|---:|
| 신용카드 | 5% |
| 현금 | 2% |

### 1.6 할인 적용 순서

1. 무비데이 할인(정률)
2. 시간 할인(정액)
3. 포인트 차감
4. 결제 수단 할인(정률)

## 2. 프로그래밍 요구 사항

- 콘솔 입출력은 구현하지 않는다.
- 모든 기능은 TDD로 구현하고 단위 테스트를 작성한다.
### 객체 지향 생활 체조 원칙을 따른다.
- 한 메서드에 오직 한 단계의 들여쓰기만 사용한다.
- else 예약어를 사용하지 않는다.
- 모든 원시값과 문자열을 포장한다.
- 한 줄에 점(.)을 하나만 사용한다.
- 축약을 지양하고 의도를 드러내는 이름을 사용한다.
- 모든 엔티티를 작게 유지한다.
- 인스턴스 변수 3개 초과 클래스를 지양한다.
- 일급 컬렉션을 사용한다.
- getter/setter/properties 사용을 최소화하고 객체가 스스로 행동하게 만든
  다.

## 도메인 설계

```

영화관(MovieTheater)
  ├─ 영화목록(MovieCatalog) [상영 가능한 영화 목록]
  │  └─ 영화(Movie) [제목, 상영시간, 상영가능기간]
  ├─ 영화스케줄(MovieSchedule) [실제 상영 편성]
  │  └─ 상영(Screening) [Movie 참조 + 시작시각 + 좌석관리]
  │     └─ 좌석관리(SeatCount)
  │        └─ 좌석(Seat)
  └─ 예매(Reservation)
     ├─ 사용자(User)
     └─ 결제(Payment)
      |└─ 할인엔진(DiscountEngine)
      |
      |_ 장바구니

```

## AI 도구 활용 기록

### 1. 활용 방식

이번 미션에서는 AI를 정답 생성 도구로 사용하기보다, 요구사항 해석과 검증을 보조하는 도구로 활용했습니다.
구현한 코드가 요구사항을 충실히 만족하는지, 빠진 부분은 없는지, 중복되거나 놓친 요소는 없는지를 교차 검토하는 용도로 사용했습니다.

도메인 구조를 검토할 때는 Screening, Seat, Cart, Payment의 경계를 중심으로 각 객체의 책임이 적절히 나뉘어 있는지 점검했습니다. 
특히 계산 책임이나 단일 컨트롤러에 과도하게 집중되지 않도록 역할을 분리하는 과정에서 AI의 도움을 받았습니다.

또한 금액 계산 결과를 비교하며 차이가 발생한 지점을 빠르게 확인하고, 오류가 발생한 부분을 효율적으로 찾는 데에도 활용했습니다.
이를 통해 문제 원인을 추적하는 시간을 줄이고, 구현 완료 시간을 좀 더 줄일 수 있었습니다.

부족한 개념이나 구현하고자 하는 방식에 대해서도 필요한 정보를 빠르게 파악하고 적용할 수 있어, 학습과 구현을 병행하는 데 도움이 되었습니다.
